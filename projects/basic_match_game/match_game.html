Basic Memory Card Game

Start Date: November 5th, 2025 - Finish Date: November 5th, 2025



Game Info

This solo project is meant to be a fun little challenge to kill some time, so to make it harder I decided to set a 24-hour limit to build a playable memory match game, aiming for something fun, functional, and clean. The goal: flip cards, find pairs, and clear the board with as few attempts as possible.



Rules / Core Mechanics

  • Mechanic 1: Generates a grid of shuffled cards
  • Mechanic 2: Allows the user to flip two cards when clicked
  • Mechanic 3: When the user flips two matching cards, they are removed from the grid


Development Plan / Concept

The game will generate a grid of flipped cards. The player will be able to click two cards at once to flip them both. If they match, they are removed. If they don't match, they're flipped back face down and the player gets to guess again. The goal is to remove all cards from the grid. The number of attempts is tracked and the goal is to finish the grid in the fewest attempts possible.

Card sprites made in Aseprite



How the game works

The first step of making this game was generating the grid, letting the user choose its size. I made a maximum grid size of 6x10 because that's what fit in the designed area and I didn't feel like scaling everything down depending on the number of cards. This is a somewhat lazy approach and if I was to continue development of this game I would change it, but since I gave myself a time limit my priorities were on other features.

Second was assigning values to the cards. I created a script in the object that stores the grid layout group, called gridManager.cs, which generates both the grid and an array of values matching its size. This script also tracks the number of matches. Once the values are generated and shuffled, it calls another script located in the card object, card.cs, which contains a cardSetup() function. This function stores the assigned value passed from gridManager.cs and uses it to assign the correct sprite using a switch statement (Circle, Star, Square, Diamond, or Triangle):
switch (value.ToLower())
{
case "circle": valueImage.sprite = circle; break;
case "star": valueImage.sprite = star; break;
case "square": valueImage.sprite = square; break;
case "diamond": valueImage.sprite = diamond; break;
case "triangle": valueImage.sprite = triangle; break;
default: break;
}

Once each card is being independently tracked, I continued making the card.cs, which contains functions that gridManager can use to interact with each individual card. These functions include OnClickRevealCard(), FlipUp(), FlipDown(), GetCardValue(), and RemoveCard().

The purpose of these functions is to allow gridManager.cs to control each and every card on the board and track their states. For example, if two cards are flipped at once gridManager.cs can stop the player from flipping a third until they are both either removed from the board or flipped back face down.

When a card is clicked, the card.cs script first checks that the card can be flipped, then if all is good it is flipped face up and card.cs lets gridManager.cs know:
public void OnClick()
{
if (!isFlipped && gridManager.CanReveal())
{
FlipUp();
gridManager.CardRevealed(this);
}
}

Initially, the RemoveCard() function destroyed the card object, but this caused the grid to shift, so instead card.cs just hides all sprites on the card object and gridManager.cs tracks the state of each card.

GetCardValue() is just a getter that returns the cardValue

FlipUp() and FlipDown() just change the sprites to visually flip the card face down or face up.

All of that, along with an if statement that checks if (matchingPairs == totalPairs), is essentially how the game works.



Comments After Project

I had a harder time than expected with the card tracking logic and at the start I didn't even consider that giving the player control over grid size could result in uneven layouts, so I added some validation to the home page where the user picks the grid size. Overall this project was pretty fun, I successfully made this game in 24 hours and because of that I'm pretty proud of it. I set the goal for 24 hours but in the end it took me around 4-5 hours of work.

In the end this project was good practice for logic in a game engine, which is extremely useful for all games that I will create in the future.
